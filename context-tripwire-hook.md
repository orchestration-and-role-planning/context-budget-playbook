# Context Tripwire — Hook Reference

A "context tripwire" is installed at the user level (every session, every project), so no agent drifts into the degraded "dumb Claude" zone. It is a four-stage governor sharing one token computation.

**What it does — four stages:**

- **WARN (>= WARN&#95;THRESHOLD, default 55,000): one calm, NON-blocking nudge.** It tells the agent it is nearing the handoff point, reassures it that it has room and should NOT rush, truncate, summarize early, or change quality, and asks only that its NEXT natural breakpoint be a tidy one (progress captured). Fires on `PreToolUse`, `PostToolUse`, and `UserPromptSubmit`, and **at most once per session** — guarded by a `.warned` marker so it never becomes a nagging alarm. The tone is deliberately anti-anxiety: a heads-up, explicitly "do NOT hand off now." On `PostToolUse`, if the jump landed in the upper half of the band (a large single-step leap), the nudge adds a measured "take stock before gathering more" line.
- **HARD (>= HARD&#95;THRESHOLD, default 75,000): forced handoff + auto-spawn. PreToolUse blocks all real work (allowing only the handoff tools), Stop/SubagentStop blocks the turn from ending, and PostToolUse injects the handoff order the instant a result crosses the line — forcing the agent to (1) write a structured handoff note, (2) spawn&#95;session a replacement with inheritModel: true, (3) write a completion marker, then stop.** The order is enforced: a completion-marker write is bounced if it is attempted before the spawn has landed, because writing the marker early flips the session to "retired" and the retired no-op would then deny the very spawn the handoff needs — stranding the work as a note with no running replacement. So the spawn must land first; the marker write then goes straight through. **Applies to all sessions, including interactive chats.**
- **WALL (>= WALL&#95;THRESHOLD, default 90,000): allow-list collapses further.** A backstop above the forced handoff for a session that somehow keeps growing past 75k: only the note write, the marker write, and spawn_session run — ToolSearch and all reads are dropped too (see isWallAction). The agent-facing message is the same calm baton-pass as the hard stop, just slightly tighter on what runs.
- **KILL (>= KILL&#95;THRESHOLD, default 120,000): final hard stop — no door, no handoff.** Far past the forced-handoff zone, a session is presumed to be producing unreliable output: a handoff note it writes can't be trusted, and a replacement it spawns would inherit the rot. So this tier offers no door and no handoff path — it denies every tool there is (spawn_session and any note or marker write included), lets the turn end, and treats the session as failed. The product owner restarts the work out of band; the hook never auto-restarts. A healthy session never reaches here — it has already handed off at 75k–90k, far below 120k — so this tier is purely a backstop against a runaway that blew past the handoff zone without finishing, and against any future token-bomb regression. It is the one deliberate exception to the "always a door, never a hard stop" principle above: this tier IS a hard stop, by design. The message still stays calm and carries no token figure — only the door framing is set aside, and only here.

**Message tone (anti-anxiety). Every message the agent reads is a calm baton-pass — being shown the door and the clean way through it — never an alarm. Deliberately absent from all agent-facing text: any ⛔ alarm marker, any raw token count, and any "limit"/"wall" wording. Token figures live only in the breadcrumb log (context-tripwire.log), never in what the agent sees. A number invites countdown math and "wall" implies a crash — both breed the context anxiety that makes an agent rush, truncate, or quietly cut quality, the very degradation the governor exists to prevent.** The internal constants stay named WARN/HARD/WALL/KILL; the agent never sees those words.

**The 20k band (55k→75k)** is wrap-up runway, not stolen quality budget. The 0→55k zone is unwarned, full quality; the nudge then lets the agent aim its next natural breakpoint to be tidy before the hard stop at 75k. Worst case (nudge ignored) is a forced handoff at 75k anyway; best case is a clean wrap-up just before it.

**In-flight growth estimate:** the token reader (`computeTokens`) returns the most recent assistant `usage` record PLUS an estimate of what has landed since. A usage record is stamped when the model generates its message, so it predates the current turn's tool results; trusting it alone let a single fan-out (many/large reads, or one huge tool output) leap a threshold invisibly between usage records. So the reader sums the bytes of every transcript line after the last usage record and adds ~bytes/4 as estimated tokens — every check reads "last known usage + what has landed since." Over-estimating is the safe direction. `PostToolUse` is wired so this re-measures the instant a result lands: at `PreToolUse` a parallel fan-out's results do not exist yet, so the estimate cannot pre-empt that one batch, but `PostToolUse` catches it as soon as the first result lands. It tightens the one-turn blind window; it does not make a single batch impossible.

**Token-reader windowing:** the reader scans the transcript tail and grows the read window ×4 (starting at 256 KB, up to the whole file) until it finds a usage record, so a single oversized trailing tool-result line can never push the latest usage record out of view and blind it.

**Handoff model ("summary + full-record backstop"):** the replacement reads the small handoff note for speed and is given the predecessor's full transcript path to consult only for missing details (never read in full). The full transcript guarantees nothing is lost.

**NO leaf exemption.** Every session is governed, including fire-and-forget leaves: a leaf that crosses a threshold gets the nudge and, at the hard limit, hands off like any other session. A redundant spawn is far cheaper than letting any session rot past the ceiling. Do NOT re-introduce a leaf carve-out.

**Files:**
- Brain: `~/.claude/hooks/context-tripwire.js` (Node; `WARN_THRESHOLD` and `HARD_THRESHOLD` constants at top, keep WARN < HARD; `computeTokens` does the measuring + estimate; `buildWarning()` holds the nudge wording; FAILS OPEN on any error — any error, unreadable file, or uncertainty returns "allow").
- On-switch: `~/.claude/settings.json` (`hooks` → `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `Stop`, `SubagentStop`, matcher `*`).
- Handoff notes land in **one folder per project** — the project root's `.handoffs/handoff_<YYYY-MM-DD_HHMM>_<shortid>.md` (readable, chrono-sortable). Completion markers go in `.handoffs/.markers/<base>.done`; the once-only warn guard is `.handoffs/.markers/<base>.warned`. Project root is found by walking up from cwd to the nearest ancestor holding a `.claude` or `.git` marker (HOME skipped), so a session in any subfolder still writes to the same place. Name base derives from the transcript's `birthtime` (stable per session → no handoff loops). Falls back to `<cwd>/.handoffs/`, then `~/.claude/handoffs/`.
- The dotted `.handoffs` name keeps these session batons distinct from a project's own "handoffs" folders (e.g. the Auditor's spec handoffs), which this hook never touches.

**Baton retention:** handoff notes + `.done` markers are RETAINED on disk for manual review (`RETAIN_BATONS = true`): the replacement reads its predecessor's note but deletes nothing, and the 24h `reap()` janitor is held off. Flip `RETAIN_BATONS` to false to restore hands-off cleanup (replacement deletes the baton once read; janitor reaps completed pairs older than 24h).

**Stop-gate escape valve:** a blocked `Stop` must never loop forever (each block is a fresh turn that only adds tokens). Each block bumps a per-session counter; once the handoff NOTE exists on disk the gate releases after `RELEASE_AFTER_BLOCKS` nudges, and `MAX_STOP_BLOCKS` is an absolute ceiling that frees even a note-less stuck agent. The worst case becomes "stop with a note written," never "spin burning tokens."

**Kill switch:** create an empty file at `~/.claude/hooks/DISABLE` to disable instantly (new invocations). Delete it to re-enable.

**Why:** automates context-handoff discipline so a long-running worker can't drift into the degraded zone. Degradation is continuous, not a cliff — quality slides as context grows, and noisy/agentic work (which BA1 is) rots fastest, so the forced stop sits inside the research-backed 60k–80k quality band (see [context-budget-model](./context-budget-model.md)), at 75k, leaving headroom under the 80k top past which tool-call noise degrades output; the calm nudge fires 20k earlier at 55k as wrap-up runway. The hook (external) does the measuring, so agents still never estimate their own context, consistent with [cross-session-context-discipline](./cross-session-context-discipline.md).

**How to apply:** to move the calm-nudge point edit `WARN_THRESHOLD`; to move the forced-handoff point edit `HARD_THRESHOLD` (keep WARN < HARD — the gap is the wrap-up runway). Edits take effect **live** — the `.js` is re-read on every hook invocation and `settings.json` is re-read by the harness mid-session.
